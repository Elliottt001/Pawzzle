# Agent 交互强化与社区闭环补全 Design

**日期：** 2026-03-20

## 背景

当前 `agent` 页已经具备基础动效，但产品主打入口和关键对话反馈仍偏弱：

- 底部输入栏按钮压感不足，主按钮不够“有存在感”
- 中间 tab 的 `agent` 入口缺少足够鲜明的品牌动效
- 聊天气泡虽然有入场，但回复仍是整段出现，缺少“浮现 + 流式输出”的生命感

同时，社区页已经存在较完整的请求逻辑与表单状态，但界面仍停留在静态 feed，导致“推荐 / 知识 / 发布”以及“发动态”未形成真实可用的交互闭环。

## 目标

在不改动后端协议和核心业务判断的前提下，一次性完成以下两类改进：

1. 强化 `agent` 页和 tab 中央入口的动效表现，让主打功能更鲜明
2. 补齐社区页的 UI-logic 接线，让现有按钮和入口真正可用

## 非目标

- 不重做整套视觉设计
- 不扩展到全局所有页面的动画系统
- 不新增依赖
- 不要求把所有历史 TODO 一次清空，只处理当前用户能直接碰到的明显断点
- 不改造成真正的服务端 SSE/WebSocket 流式协议，本轮先做前端渐进式流式呈现

## 方案对比

### 方案 A：只增强局部按压动画

优点：
- 改动最小
- 风险低

缺点：
- 解决不了回复“整段蹦出”的僵硬感
- 也无法修复社区“按钮有状态、界面没闭环”的核心问题

### 方案 B：重做一套全局交互动效系统

优点：
- 统一性最好

缺点：
- 远超本轮需求
- 会显著扩大改动面和验证成本

### 方案 C：局部强化主路径体验，并补齐高价值逻辑闭环

优点：
- 直接命中用户感知最强的问题
- 改动边界仍然可控
- 能把“主打功能”和“明显断点”一起补齐

缺点：
- 需要同时触达 `agent`、tab bar 和 `community` 三处

**采用方案：** C

## 设计

### 1. Agent 页动效

#### 1.1 底部输入栏按钮

- 在现有 `AnimatedPressable` preset 基础上区分 `icon`、`text`、`hero-icon`
- 语音按钮与发送按钮增强为更明显的按下缩放、轻微上浮和更强回弹
- 禁用态保持清楚，但不会误导为“可点击”

#### 1.2 对话气泡

- 保留气泡入场的浮现感
- AI 回复新增“逐字输出”层：
  - 新消息先挂载一个空内容气泡
  - 再按固定节奏追加文本，模拟流式输出
  - 用户消息与调试消息仍保持即时显示
- 等待文案只作为等待态，不再替代真正回复的呈现节奏

#### 1.3 阶段切换

- 继续使用轻量淡入上浮
- 不把开始页也卷入统一过场，避免节奏被拖慢

### 2. 中央 Agent Tab 按钮

- 给 `PawzzleTabBar` 中的中间按钮增加独立动效层
- 聚焦当前 tab 时呈现更明显的抬升、发光/脉冲氛围
- 按压时提供更强的弹性反馈，明显区别于普通 tab

### 3. 社区页闭环

#### 3.1 页面结构

- 恢复并接上线内已有的三段式内容：
  - `推荐`：动态 feed
  - `知识`：后端返回的 guides
  - `发布`：发布卡片 / 动态 / 指南
- 搜索框驱动推荐内容过滤

#### 3.2 发布功能

- 让 `uploadOpen`、`uploadType`、`handleCreateCard`、`handleCreateContent` 真实接入界面
- 至少保证：
  - 发动态可用
  - 发指南可用
  - 发宠物卡可走完表单和提交

#### 3.3 状态反馈

- 加上加载、成功、错误提示
- 未登录时给出明确引导，不出现“点了没反应”

## 文件边界

- Modify: `frontend/app/(tabs)/agent.tsx`

- Modify: `frontend/components/agent/motion.ts`
  - 扩展动效 preset，覆盖更强的主按钮与 tab 中心按钮
- Modify: `frontend/components/agent/motion.test.ts`
  - 为新增 preset 合约补测试
- Modify: `frontend/components/pawzzle-tab-bar.tsx`
  - 中央 tab 主按钮强化动画
- Modify: `frontend/app/(tabs)/community.tsx`
  - 接上已有逻辑，补齐推荐 / 知识 / 发布闭环

## 测试策略

- 先为 `motion.ts` 新增 failing tests，锁定更强按钮 preset 的 contract
- 为社区页提取少量纯函数后补最小 Node 测试，覆盖 tab/filter/form gate 等容易回归的逻辑
- 针对修改文件运行定向 ESLint
- 运行前端全量 lint，区分本次新增问题与仓库历史 warning

## 风险与控制

### 风险 1：流式输出导致消息重复或重播

控制：
- 只对 AI 新回复走流式层
- 用稳定 message id 驱动状态
- 在新请求开始和完成时明确清理 timer

### 风险 2：社区页改动过大引起样式或逻辑串线

控制：
- 尽量复用文件中已有状态和样式
- 如需新增 helper，优先提纯逻辑而不是大拆 UI

### 风险 3：主 tab 按钮动画过猛影响可用性

控制：
- 聚焦态用柔和脉冲，按压态才强调弹性
- 避免持续大幅位移和跳动

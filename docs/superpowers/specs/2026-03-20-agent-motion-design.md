# Agent 页面动效设计

## 概述

本次改动要提升 Expo `agent` 页的前端交互质感，解决当前“按钮点击太硬、对话气泡突然出现、阶段切换缺乏衔接”的问题。

整体动效方向为“克制但明显更灵动”，重点覆盖：

- `agent` 全流程内按钮的点击反馈
- 用户与 agent 对话气泡的入场动画
- `quiz`、`chat`、`survey`、`result` 四个阶段之间的轻量过渡

本轮改动范围限定在 [`frontend/app/(tabs)/agent.tsx`](/home/yuuray/桌面/Pawzzle/frontend/app/(tabs)/agent.tsx)，不扩展到其他页面。

## 目标

- 在不改变页面信息结构和业务逻辑的前提下，让 `agent` 页手感更顺、更完整。
- 为开始页、问卷页、聊天页、结果页中的主要交互按钮提供统一的点击反馈。
- 为新增聊天消息提供自然的入场动效，让对话更像“出现”而不是“硬切”。
- 为阶段切换提供轻微的衔接感，减少页面状态跳变时的生硬感。
- 保持现有请求时机、数据流、推荐逻辑与导航逻辑不变。

## 非目标

- 不重做 `agent` 页的布局、文案或流程结构。
- 不做 `agent` 页之外的全站按钮动效统一。
- 不做明显的整屏飞入飞出、横滑切屏或夸张弹跳。
- 不涉及后端接口、接口协议或推荐策略调整。

## 现状与约束

当前 `agent` 页已经在开始页使用 React Native `Animated` 实现吉祥物漂浮效果，但其余大部分按钮仍然使用静态 `Pressable`，只有简单缩放按压态或完全没有动效。聊天消息在追加后会直接渲染出来，因此整体体验偏“硬”。

当前前端已经安装 `react-native-reanimated`，因此本次采用混合方案：

- 保留开始页已有的 `Animated` 吉祥物漂浮效果
- 新增的按钮反馈、气泡入场、阶段切换统一使用 `react-native-reanimated`

这样可以避免为了统一技术栈而重写现有稳定代码，同时给新增动效提供更细腻的弹性、透明度与位移动画控制。

## 推荐方案

采用“保留旧 `Animated`，新增交互层用 `Reanimated`”的混合方案。

### 选择原因

- 保留已经稳定的开始页吉祥物漂浮逻辑，降低改动风险。
- 把新增动效集中在当前文件内完成，避免大拆页面结构。
- `Reanimated` 更适合做按钮弹性、气泡入场和阶段淡入这类轻量但细腻的交互效果。
- 能满足“克制但明显更灵动”的目标，不需要引入额外依赖。

## 动效语言

整体气质应当偏温和、柔软、产品化，而不是夸张、卡通或强表演感。

### 原则

- 动效时间短，反馈明确，不拖沓
- 用轻压、回弹、上浮表达层次，不用大幅位移
- 按钮反馈与内容入场有区分，但节奏统一
- 动效只辅助理解层级，不喧宾夺主

### 建议时长

- 按钮点击反馈：约 `140ms` 到 `220ms`
- 聊天气泡入场：约 `220ms` 到 `320ms`
- 阶段切换入场：约 `180ms` 到 `260ms`

### 具体表现

- 主要按钮：按下时略微压缩，松开时柔和回弹，可叠加轻微透明度或阴影变化
- 问卷选项卡：点击时短促压下，选中态的底色和边框平滑切换，不做明显弹跳
- 用户消息：从右下方较短距离滑入并淡入，节奏更利落
- agent 消息：从下方轻微上浮并淡入，节奏稍柔和
- 阶段切换：使用淡入加轻微上浮，不做整屏横滑

## 范围

本次明确包含以下节点：

- 开始页 CTA 按钮
- 问卷选项卡
- Survey 提交按钮
- Survey 跳过按钮或跳过文本动作的轻量点击反馈
- 聊天发送按钮
- 聊天语音按钮
- 结果页“重新测试”按钮
- 用户与 agent 聊天气泡入场
- `quiz`、`chat`、`survey`、`result` 四阶段内容的轻量进入过渡

本次不包含：

- `agent` 页之外的按钮
- 开始页吉祥物之外的新增大型装饰动画
- 重写聊天布局或消息结构

## 组件设计

实现尽量收敛在 [`frontend/app/(tabs)/agent.tsx`](/home/yuuray/桌面/Pawzzle/frontend/app/(tabs)/agent.tsx) 内，以轻量局部封装为主，而不是抽出一套大而全的全局动画系统。

### `AnimatedPressable`

为 `agent` 页中的按钮与可点击卡片提供统一的轻量交互包装。

职责：

- 处理 `press in` 与 `press out` 的缩放动画
- 视按钮类型附加透明度、轻微位移或阴影层级变化
- 正确处理禁用态，避免禁用时仍然表现得像可操作
- 同时支持大按钮、文本按钮和图标按钮

主要使用位置：

- 开始页 CTA
- 问卷选项卡
- Survey 提交按钮
- Survey 跳过动作
- 聊天发送按钮
- 聊天语音按钮
- 结果页“重新测试”

### `AnimatedChatBubble`

为聊天消息提供只在“新消息出现时”触发的入场动效。

职责：

- 根据消息角色区分用户消息与 agent 消息的入场方式
- 使用透明度与位移完成轻量入场，不改动现有布局关系
- 避免因为输入框变化、等待文本轮换或其他重渲染而重复播放

角色差异：

- 用户消息：位移更短、速度更快、方向偏右下到当前位置
- agent 消息：位移略长、节奏更柔和、方向偏下到当前位置

### `AnimatedPhaseView`

为 `quiz`、`chat`、`survey`、`result` 四个阶段内容提供统一的轻量入场容器。

职责：

- 当阶段变化时触发一次进入动画
- 使用淡入和轻微上浮建立页面衔接感
- 不做复杂离场动画，避免阶段切换快时出现闪烁和残留

应用位置：

- 问卷内容区
- 聊天内容区
- Survey 内容区
- 结果页内容区

说明：

- 开始页保持当前结构，不额外加入整屏切换动效
- 开始页的重点改动在 CTA 点击反馈，而不是页面级转场

## 状态与数据流

本次改动只增强渲染层，不改变现有业务状态语义。

当前关键状态：

- `hasStarted`：控制是否展示开始页
- `phase`：控制 `quiz`、`chat`、`survey`、`result`
- `messages`：驱动聊天列表渲染
- `status`、`evaluation`、录音与识别状态：驱动按钮禁用与等待提示

动效应当从这些状态派生，但不能改变它们的判断逻辑。

### 按钮交互流

- 现有点击回调仍然是唯一真实触发源
- 动效只发生在 UI 层，不参与业务判断
- 禁用态优先级高于动效，按钮不能因为回弹而看起来像“操作成功”

### 聊天消息流

- 新消息被追加到 `messages`
- 对应气泡渲染
- 气泡基于消息身份触发一次入场动画
- 已经显示的旧消息保持稳定，不重复进场
- 等待中的 loading 气泡如果以新节点挂载，可以使用同类但更轻的入场效果

### 阶段切换流

- 阶段状态仍按现有逻辑流转
- 新阶段内容容器在挂载后执行一次轻量进入动画
- 不改变请求发起时间、总结生成时间和推荐结果生成时机

## 异常与边界情况

- 动效不能阻塞网络请求、输入提交、录音收起或阶段切换。
- 当 `status` 标记为忙碌或按钮被禁用时，交互限制必须优先于动效表现。
- 快速切换阶段时，不应出现旧内容残留、闪烁或连续重放。
- 气泡入场不能因为输入框输入、等待文案轮换或普通 rerender 被重复触发。
- 如果设备性能一般，动效也应自然退化；优先使用短时长的透明度和 transform，避免重布局类动画。

## 可用性要求

- 动效不能让用户等待内容可用，内容应在短时间内进入稳定状态。
- 禁用态与可点击态必须依旧清晰可辨。
- 加入动画包装后，文本可读性、间距和触控热区不能被破坏。
- 用户即使不特别关注动画，也应能感受到“页面更顺了”。

## 技术实现约束

- 保留 `AgentStartScreen` 里现有的 `Animated` 漂浮动画。
- 新增动效统一优先使用 `react-native-reanimated`。
- 默认先在同文件内组织轻量 helper；只有在结构明显失控时才考虑抽文件。
- 不新增第三方依赖。
- 不修改 API helper、语音识别逻辑和推荐逻辑。

## 验证策略

这次改动以 UI 质感提升为主，但仍需提供可验证证据。

### 自动化验证

- 如果在不引入额外复杂度的前提下能为新增 helper 补最小可行测试，则优先补充
- 至少保证该文件通过 TypeScript 与 lint 校验

考虑到当前前端工程没有正在使用中的完整页面测试链路，本次可以以 lint 和针对性交互验证作为主要完成依据。

### 手动验证清单

1. 开始页 CTA 点击时有柔和压下与回弹感。
2. 问卷选项卡点击反馈自然，选中态变化清晰。
3. 从问卷进入聊天时，页面过渡比当前更连贯但不过度抢戏。
4. 用户消息和 agent 消息入场节奏有区分，但整体风格统一。
5. 发送按钮和语音按钮在可用、不可用、忙碌状态下反馈正确。
6. Survey 提交与跳过动作的手感与其他按钮一致，不显得突兀。
7. 结果页进入与“重新测试”按钮点击反馈不再生硬。

## 风险

- 在同一文件混用 `Animated` 与 `Reanimated`，如果边界不清晰，容易造成实现风格不一致。
- 如果消息入场和 rerender 绑定错误，聊天区可能会频繁重复播放动效，影响阅读。
- 如果阶段切换做得过重，会让整个流程显得更慢、更不稳。

## 风险缓解

- 把旧 `Animated` 严格限制在现有吉祥物漂浮效果中，其余新增动效统一交给 `Reanimated`。
- 让气泡动画绑定在消息 `id` 或稳定身份上，而不是普通渲染次数上。
- 把阶段切换控制在“单向轻入场”，不用复杂离场配对动画。

## 可进入实现计划的判断

该规格已经具备进入 implementation plan 的条件：

- 范围只覆盖一个页面，没有跨多个独立子系统
- 包含了目标、非目标、组件拆分、状态边界、风险与验证方式
- 对“哪些要动、怎么动、哪些不动”给出了足够清晰的约束
- 后续实现可以直接围绕本文件中的局部动效层展开，无需额外补需求
